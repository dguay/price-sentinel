1. Executive Summary

What this project is. Price Sentinel is a local, dependency-free Ruby CLI (stdlib only) that monitors explicit product pages against target prices. It validates a YAML config, scans enabled sources with deterministic extractors, classifies each result (found / no_match / uncertain / blocked / error), prepends a scan block to a Markdown log, writes local state (last scan, alert dedupe, scan lock), and sends ntfy notifications. Scheduling is deliberately external. An agent skill (.codex/skills/price-sentinel/SKILL.md, symlinked into .claude/skills/) defines the agent-facing workflow contract. First real use case: Apple Canada MacBook monitoring.

Current maturity. Solid v1. Clean module boundaries, disciplined domain language (CONTEXT.md), honest safety posture (ADR-0001: never bypass blocked sources), and a genuinely good end-to-end test suite (63 tests covering CLI workflows, locking, dedupe, fixtures for real Apple pages). Documentation quality is above average for a project this size. There is no CI, no Ruby version pin, and the codebase runs on EOL Ruby 2.6.

Biggest risk. Scan robustness against real-world network behavior. Two crash paths are confirmed or near-certain: (a) an enabled check/source without an id passes validation but crashes the scan with a raw KeyError stack trace (reproduced during this audit; still present at 271b14d); (b) an SSL handshake failure (or EOFError/Net::HTTPBadResponse) on any single source raises an unrescued exception that aborts the entire scan — no Markdown log, no state, no notifications for the sources that succeeded. For an unattended scheduled monitor, "one bad source kills the whole run" is the failure mode that matters most. Commit 271b14d grew the extraction surface ~60% (613→997 lines in source_extractors.rb) without widening these rescue paths, so the exposure increased. A close second, new in 271b14d: the free-form retailer label now silently alters extraction semantics — including a heuristic that divides prices by 100 for reebelo_ca — with no documentation or validation (DATA-017).

Best next move. Harden the scan loop: make per-source failure isolation airtight (rescue StandardError per source, widen extractor rescue lists), make id required by validation, and follow HTTP redirects in normal scans. All three are small, test-backed changes that directly protect the core daily workflow.

Highest-leverage recommendations:

  1. Isolate per-source failures (BUG-001) — one flaky source must never lose an entire scan's results and state.
  2. Require id on enabled checks/sources at validation (BUG-002, confirmed crash) — validation currently promises more than it delivers.
  3. Follow redirects during scans (BUG-003) — retailer URLs redirect routinely; diagnosis already follows them, scans do not.
  4. Fix notification failure semantics (BUG-005) — a single failed ntfy POST currently aborts state writes and loses last-scan state.
  5. Add CI + Ruby version pin (TEST-011, SEC-010) — the excellent test suite is worth nothing if it only runs when someone remembers, on an EOL interpreter.
  6. Document and validate retailer-keyed behavior (DATA-017, new in rev. 2) — a typo in a free-form label must not be able to change a price by 100×.

3. Current Feature Assessment

Implemented and working

  • Config validation with clear per-field errors, draft (disabled) check/source tolerance, transport validation.
  • Scan with strict hit classification: price ≤ target, currency match, required constraints (condition/seller/availability/ships_to), attribute identity matching. Well tested, including "low price but wrong product never hits" cases.
  • Two real extractors: a now-substantial generic extractor (JSON-LD, OG meta, embedded application/json, Shopify product JS blobs, HTML search-result tiles; candidate selection against expected_attributes; blocked-body and Amazon.ca-503 detection; explicit-zero-results and mismatch → no_match) and the Apple Canada extractor — both verified against saved real-page fixtures.
  • product_url extraction (new in 271b14d): hit lines in the Markdown log now link the specific matched product page, not just the configured source URL.
  • Markdown log with run-ID keyed block replace/prepend semantics, tested for creation, replacement, ordering, and write-failure reporting.
  • Concurrency lock (flock + payload + stale recovery), scoped per config path, tested cross-process.
  • ntfy notifications with category policy, hit dedupe (first hit / price drop / re-entered hit), templating, bearer token via env var. Tested against a local HTTP server.
  • Source diagnosis with structured/visible evidence and optional HTML artifact saving.
  • Starter templates + init-config with overwrite refusal.

Partial

  • Error resilience of live scans — the happy path and several curated failure paths (HTTP status codes, markdown write failure) are handled, but real-world network failure classes (SSL, EOF, malformed responses, redirects) are not (BUG-001, BUG-003).
  • no_match state — no longer dormant: since 271b14d the generic extractor emits it (candidate mismatch, zero-result search pages). But the Markdown log has no "No Match" group and the summary counts don't include it, so these results are invisible in the log — promoted to finding UX-019.
  • Retailer support — the generic extractor now special-cases six retailers (amazon_ca, jumpplus, owc_macsales, newegg_ca, cdw_ca, reebelo_ca) via the free-form retailer label, none of which is documented in the README config reference (DATA-017).
  • Attribute extraction — MacBook-oriented heuristics (chip, memory, storage, screen size) over full-page visible text; works on the fixtures but structurally prone to cross-product contamination on listing pages (DATA-008).

Documented but not implemented

  • explain-results is listed as a Skill Command (README line 48, SKILL.md) but has no CLI subcommand — it is agent-side interpretation of last-scan.json. This is intentional per the skill/CLI split, but the README's Skill Commands section does not mark it as CLI-less the way it documents the other four; a first-time reader may try bin/price-sentinel explain-results.
  • CONTEXT.md mentions "LLM-assisted Playwright inspection ... reserved for diagnostics" — no Playwright integration exists anywhere. Aspirational vocabulary only.

Most important user workflow

The scheduled Daily Scan: validate → scan → read Markdown log → get ntfy pushed only for meaningful events. Everything that threatens unattended reliability of that loop (crash on one source, false lockouts, lost state after a notification hiccup) is the highest-value work. The interactive workflows (init-config, diagnose-source) are secondary and in good shape.

BUG-001One failing source can crash the entire scan (unrescued network exception classes) BugHighConfidence: High
Evidence
Re-verified open at 271b14d. lib/price_sentinel/source_extractors.rb:109 and :752 — both live extractors rescue only KeyError, URI::InvalidURIError, SocketError, SystemCallError, Timeout::Error, Net::OpenTimeout, Net::ReadTimeout. Missing: OpenSSL::SSL::SSLError (not a SystemCallError), EOFError/IOError (server closes connection mid-response), Net::HTTPBadResponse, Net::ProtocolError. By contrast, source_diagnosis.rb:82 rescues IOError and OpenSSL::SSL::SSLError — the author already knows these occur. The second line of defense, Scanner.scan_source (lib/price_sentinel/scanner.rb:130), rescues only ArgumentError, so anything the extractor misses escapes scan_file entirely. 271b14d made this worse in practice: the generic extract path now runs far more parsing code (candidate selection, JS-blob regexes, JSON.parse of embedded blobs, URI.join in absolute_url — which rescues URI::InvalidURIError but not ArgumentError from nil bases) before returning, all inside the same too-narrow rescue.
Why it matters
A TLS misconfiguration or dropped connection on one source aborts the whole scan run: no Markdown log block, no last-scan state, no notifications — including for sources that succeeded. The user sees a raw stack trace in scheduler logs. For an unattended daily monitor this silently creates gaps in the log history.
Recommended fix
Two layers: (1) add the missing exception classes to both extractor rescue clauses; (2) broaden Scanner.scan_source's rescue from ArgumentError to StandardError so any future extractor bug degrades to a per-source error result instead of a crash. Keep the message format ("#{e.class}: #{e.message}") so diagnosis stays possible.
Step-by-step execution plan
  1. Open lib/price_sentinel/source_extractors.rb. In ProductPageExtractor.extract (rescue at line 109) change the rescue list to: KeyError, URI::InvalidURIError, SocketError, SystemCallError, IOError, Timeout::Error, Net::OpenTimeout, Net::ReadTimeout, Net::ProtocolError, OpenSSL::SSL::SSLError. (EOFError < IOError, Net::HTTPBadResponse < Net::ProtocolError — covered.)
  2. Add require "openssl" at the top of source_extractors.rb (currently not required there).
  3. Apply the identical rescue-list change to AppleCanadaProductPageExtractor.extract (rescue at line 752).
  4. Open lib/price_sentinel/scanner.rb. Change line 130 from rescue ArgumentError => e to rescue StandardError => e. Note the rescue block itself calls check.fetch("id")/source.fetch("id") — that is addressed by BUG-002; if implementing this finding alone, use check["id"]/source["id"] inside the rescue block to avoid a rescue-within-rescue crash.
  5. Add tests (below), run the full suite.

BUG-004False "Scan already active" for up to 30 minutes after a crashed scan BugMediumConfidence: HighDONE
Evidence
Re-verified open at 271b14d (monitor_state.rb untouched). MonitorState.acquire_lock (monitor_state.rb:100-123): flock is acquired non-blocking; then, if a lock payload exists and is not older than stale_lock_after_ms (default 30 min), it raises ActiveScanError — even though the successful flock already proves no live process holds the lock (flock is released by the OS when a process dies). A scan killed with SIGKILL (or a machine sleep/crash) leaves the payload file behind; every scan attempt in the next 30 minutes is refused with "Scan already active" although nothing is running.
Why it matters
For a daily scan this is invisible; for hourly or agent-retried scans it blocks recovery exactly when a previous run just failed — the moment you most want the next run to proceed. The message is also misleading ("already active" when nothing is).
Recommended fix
Trust flock for liveness. Since the payload records pid (monitor_state.rb:172), add a liveness check: if the payload's pid is not alive (Process.kill(0, pid) raising Errno::ESRCH), treat the payload as stale regardless of age and recover (existing "Recovered stale scan lock" path). Keep the time-based threshold as the fallback for edge cases (pid reuse, payloads without pid). Note: flock success alone is arguably sufficient, but the payload check guards NFS-like setups and same-process re-entry; the pid check is the minimal correct tightening.
Step-by-step execution plan
  1. In monitor_state.rb, add helper process_alive?(pid): returns false for nil/non-integer; Process.kill(0, Integer(pid)); true rescuing Errno::ESRCH → false, Errno::EPERM → true, ArgumentError/TypeError → false.
  2. In acquire_lock, change the refusal condition at line 110 to: if existing_payload && process_alive?(existing_payload["pid"]) && !stale_lock_payload?(existing_payload, stale_lock_after_ms(config)).
  3. Ensure the "Recovered stale scan lock" warn still fires when recovering via dead-pid (it will — existing_payload is truthy).

Re-verified open at 271b14d (notifications.rb and bin untouched). Notifications.post_ntfy raises NotificationError on any non-2xx or transport exception (notifications.rb:135-156). deliver has no rescue around the candidate loop (:43-47), so the first failure skips: remaining candidates, remaining transports, and write_state (:48). Three concrete effects: (1) last-scan.json — the machine-readable record the skill's explain-results workflow depends on — goes stale even though the scan succeeded; (2) notifications already sent before the failure are re-sent on the next run (state never recorded), while candidates after the failure were never attempted — both duplicate and dropped alerts from one incident; (3) with multiple transports, one broken transport silences all others.

Make delivery best-effort per candidate/transport: collect failures instead of raising, always write alert state for what was actually sent, return failures in the summary, and have bin print them as warnings with exit 0 (scan itself succeeded) — or exit 1 after writing last-scan state, per owner preference (see §9 Q2).   3. In deliver, wrap the per-transport POST: begin; post_ntfy(...); sent += 1; rescue NotificationError => e; failures << e.message; end (restructure the current sum into an explicit loop with sent/failures counters).

ARCH-006Price-drop dedupe compares against last scan price, not last notified price — slow drift never notifies ArchitectureMediumConfidence: HighDONE
Evidence
Re-verified open at 271b14d. next_state (notifications.rb:239-256) overwrites previous["price"] with the current price on every scan, notification or not. hit_reason's price_drops path (:84) and price_drop? (:113-125) therefore compare each scan to the immediately previous scan. With price_drop_threshold.amount: 5, a price falling $4/day for a week (total $28 drop) never crosses the per-scan threshold → zero notifications, despite the threshold's documented intent ("Minimum drop required for a price_drops notification").
Why it matters
The threshold exists to say "tell me when it gets meaningfully cheaper". As implemented it means "tell me when it gets meaningfully cheaper between two consecutive scans" — with frequent scans this converges to never notifying. Silent under-alerting is the worst failure mode for an alerting tool.
Recommended fix
Track last_notified_price per source in alert state. price_drop? compares current price to last_notified_price (falling back to current stored price for pre-migration state). Set last_notified_price only when a hit notification for that source is actually sent. Old state files lack the field — first comparison falls back gracefully, no migration needed.

  2. Change next_state(state, report) to next_state(state, report, notified_hit_keys); when building each source entry, set "last_notified_price" => notified_hit_keys.include?(key) ? result.price : previous["last_notified_price"].
